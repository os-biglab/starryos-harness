# 反幻觉协议

## 7 级验证体系

只报告 1-5 级证据。6-7 级不可报告，必须先升级到 1-5 级。

| 级别 | 名称 | 可报告 | 方法 |
|------|------|--------|------|
| 1 | 可执行测试 | ✅ | 写测试，运行，观察不同结果 |
| 2 | 源码级证明 | ✅ | 代码中可见的错误 (错误变量、错误条件) |
| 3 | 属性违反 | ✅ | 可测量的不变量被打破 (如锁序环) |
| 4 | 差异分析 | ✅ | 同一代码在不同配置下行为不同 |
| 5 | Linux 源码对比 | ✅ | StarryOS 代码与 Linux 实现对比，发现语义差异 |
| 6 | 静态模式 | ❌ | grep/regex 匹配 (可能是误报) |
| 7 | LLM 推理 | ❌ | "我觉得这里可能有问题" (无证据) |

## 升级路径

当怀疑一个问题但只有 6-7 级证据时:

```
怀疑 (7级) → 写测试验证 (1级) → 确认/否认
怀疑 (7级) → 读源码找到具体错误 (2级) → 确认/否认
怀疑 (7级) → 运行属性检查工具 (3级) → 确认/否认
```

## 7 种常见幻觉陷阱

### 1. "这个看起来可能会竞争"
**陷阱**: 看到共享变量就怀疑 data race，但没有检查是否有锁保护。
**正确做法**: 枚举所有访问点，检查每个点的锁保护。

### 2. "这个函数没有处理 X 情况"
**陷阱**: 假设某路径不会被触发，但没有追踪调用链。
**正确做法**: grep 所有调用者，确认输入范围。

### 3. "Linux 做了 Y 所以 StarryOS 也应该做 Y"
**陷阱**: 假设 Linux 的所有行为都是必要的，但 StarryOS 可能有不同的设计选择。
**正确做法**: 确认这个差异是否导致了实际问题。

### 4. "这个 unsafe 块不安全"
**陷阱**: 看到 unsafe 就标记为问题，但没有验证实际的 safety 条件。
**正确做法**: 检查 SAFETY 注释，验证 preconditions 是否满足。

### 5. "这个锁的粒度太粗"
**陷阱**: 主观判断锁的粒度，但没有测量实际的争用情况。
**正确做法**: 测量锁争用，如果有性能问题再优化。

### 6. "这个错误处理不够优雅"
**陷阱**: 将风格偏好标记为 bug。
**正确做法**: 只报告会导致实际问题的错误处理问题。

### 7. "这个 API 设计不好"
**陷阱**: 将设计偏好标记为缺陷。
**正确做法**: 只报告会导致 bug 或违反现有约定的 API 问题。

## 评审时的应用

在 `os-review` 技能中，每个发现必须:

1. 声明验证级别 (1-5)
2. 提供具体证据 (文件:行号、测试输出、对比结果)
3. 如果只有 6-7 级证据，先升级再报告

```
## Finding: Potential data race in run_queue.rs:123

**Verification level**: 2 (Source-level proof)
**Evidence**: Line 123 accesses `self.current` without holding `self.lock`.
  The lock is acquired at line 115 but released at line 118 via `drop(guard)`.
  Line 123 is outside the lock scope.

**Impact**: SMP=2 时可能导致 task 调度错误。
**Fix**: 将 line 123 移入锁保护范围。
```

## 工具辅助

使用确定性工具提升验证级别:

| 工具 | 提升 |
|------|------|
| `lock-order-graph.py` | 7级 → 3级 (锁序环检测) |
| `pattern-scanner.py` | 7级 → 6级 (静态模式，需人工确认) |
| `abi-check.py` | 7级 → 5级 (Linux 对比) |
| 运行测试 | 7级 → 1级 (可执行证据) |
